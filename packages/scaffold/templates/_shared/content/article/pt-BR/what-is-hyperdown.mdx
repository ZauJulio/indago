# O que é o HyperDown?

**Quer busca full-text de verdade no seu site sem subir um único serviço de backend?** É essa
a proposta inteira. Você escreve Markdown; o HyperDown te entrega uma camada de conteúdo
tipada e pesquisável que vai _dentro do seu build_ e não custa nada para rodar.

O **HyperDown** (`@indago/hyper-down`) transforma uma pasta de arquivos Markdown/MDX
numa camada de conteúdo tipada e com busca full-text — **sem serviço de banco de dados,
sem API e sem servidor de CMS**. No build, ele compila o **frontmatter** do seu conteúdo
num índice **FTS5** SQLite compacto e contentless; em tempo de requisição, os loaders de
rota do servidor consultam esse índice com `bun:sqlite` (ou `node:sqlite` no Node ≥ 22,
como na Vercel). O navegador nunca vê um banco de dados.

É um dos dois motores independentes do toolkit **Indago**. O HyperDown cuida da **prosa
em Markdown/MDX**; seu irmão [HyperJson](/pt/articles/what-is-hyperjson) cuida do
**JSON estruturado**. Eles não compartilham código — adote um ou os dois.

## Por que o HyperDown?

A maioria dos "sites de conteúdo" recorre a um CMS headless, a um banco de dados hospedado ou
a um SaaS de busca — três peças móveis para provisionar, pagar e proteger só para publicar
alguns artigos. O HyperDown elimina essa camada inteira. Em troca, você ganha:

- **Busca full-text de verdade, zero infraestrutura.** Busca FTS5 por prefixo em títulos,
  descrições **e** corpos — rodando a partir de um arquivo SQLite que vai dentro do seu build.
  Sem Algolia, sem Elasticsearch, sem linha gerenciada parada na sua fatura.
- **Seu conteúdo continua no git.** Artigos são só arquivos `.mdx` numa pasta: revise em pull
  requests, compare diffs, crie branches, reverta — o fluxo que seu time já tem.
- **Tipado de ponta a ponta, sem desvio.** O frontmatter vira uma interface TypeScript gerada,
  então um campo renomeado quebra o build, não a produção.
- **Um banco pequeno e autocontido.** O índice FTS é _contentless_ — guarda tokens e
  metadados, nunca a prosa — então o `.db` fica pequeno e viaja junto com o seu deploy. Nada
  para hospedar.
- **Pronto para agentes.** Um servidor MCP embutido deixa o Claude Code ou o opencode criar
  tipos de conteúdo e escrever artigos por você. (Este post foi criado assim.)
- **Uma camada, quatro frameworks.** Um único `bun create @indago/app` roda a mesma camada de
  conteúdo em Vike, React Router v7, TanStack Start ou Next.js.

Se você publica Markdown e já pensou _"eu só queria busca sem rodar um servidor para isso"_ —
isto foi feito para você. O resto do post mostra como ele faz isso acontecer.

## A ideia central: metadados no SQLite, corpo nos globs

O HyperDown divide cada conteúdo em duas partes:

- **Os metadados do frontmatter** (título, tags, data…) são parseados e persistidos no
  SQLite — um `.db` por tipo de conteúdo. É contra isso que busca, filtros, facetas e
  ordenação rodam.
- **O corpo MD/MDX nunca é guardado no SQLite.** Ele é compilado para um componente
  React pelo `@mdx-js/rollup` e carregado por um mapa **estático de
  `import.meta.glob`** que o codegen escreve na árvore `.hyper-down/` do seu app.

As tabelas FTS5 são **contentless** (`content=""`): guardam apenas o índice invertido,
nunca o texto original. O corpo **é tokenizado nesse índice** — então a busca full-text
alcança a sua prosa — mas nunca é persistido, o que mantém o `.db` minúsculo e o
invariante "corpo nunca guardado" intacto. `tags` e `categories` recebem tratamento
extra: são guardadas como arrays JSON na tabela de metadados, achatadas no índice FTS e
espelhadas numa tabela-ponte indexada `<type>_tags`, para que filtros de tag e contagens
de faceta continuem sargables (nunca um scan com `LIKE`).

## Arquitetura: três passos de build, dois caminhos de runtime

No build (o `buildStart` do plugin Vite, o comando `hyperdown gen:db` da CLI, ou o
adapter de Next.js):

1. **O codegen** escreve de forma idempotente na árvore `.hyper-down/` do app: uma
   interface ambiente `<Type>Meta` (`types.ts`), um **proxy preguiçoso
   `<type>Repository`** (`builder.ts` — o `new` é adiado para depois da avaliação do
   módulo, o que evita uma corrida de ordem de inicialização de chunks no SSR
   bundleado), um mapa **estático e eager** de `import.meta.glob` dos corpos MDX
   (`modules.ts`) e um barrel `default.ts` exportando `contentModules`.
2. **O writer** parseia o frontmatter de todos os arquivos num pool paralelo de
   leitura/parse/validação e persiste em série numa única transação — emitindo um `.db`
   por tipo de conteúdo: a tabela de metadados, a ponte `<type>_tags` e a tabela FTS5
   contentless.
3. No `closeBundle`, cada `.db` é copiado para `dist/metadata/`, deixando a saída do
   build **autocontida** — um deploy SSR lê os bancos de lá sem precisar da árvore
   `content/` de origem.

No runtime, os caminhos são estritamente separados:

- **Servidor (loaders de rota)** — `ContentRepository<T>`, exportado apenas de
  `@indago/hyper-down/server`: `search()` (FTS5 `MATCH` + filtros + ordenação +
  paginação), `distinctValues()` (facetas para UIs de filtro) e `getMetaBySlug()`
  (metadados serializáveis com fallback de idioma). O match FTS roda **em todos os
  idiomas** e é mapeado de volta para slugs — "slow" e "lenta" encontram o mesmo
  artigo, uma linha por slug no idioma pedido. O `.db` abre somente-leitura do disco,
  memoizado entre requisições, com cache de prepared statements.
- **Cliente (views)** — `createContentResolver(contentModules[type])` retorna
  `getContent(slug, lang)`, que resolve o componente MDX preguiçoso; renderize com
  `MdxRender`. Esse caminho não toca em código de banco, então os built-ins `node:*`
  nunca chegam ao bundle do navegador.

Um loader real de um app gerado — as listagens são **dirigidas pela URL** (`?q`,
`?tag`, `?page`, `?sort`), então o servidor refaz a consulta a cada navegação:

```ts
const [result, tags] = await Promise.all([
  articleRepository.search({
    locale,
    searchQuery,
    filters: activeTag ? { tag: activeTag } : {},
    sort: { sortBy: "date", sortDir: "desc" },
    pagination: { page, pageSize: 6 },
  }),
  articleRepository.distinctValues({ isJson: true, column: "tags", sortByFrequency: true }, locale),
]);
```

## Estrutura do projeto: o que você escreve, o que é gerado

Um app consumidor mantém conteúdo, config e artefatos gerados lado a lado:

```text
meu-app/
├── content/
│   └── article/                coleção HyperDown
│       ├── en/hello.mdx        pastas por idioma; slug = nome do arquivo
│       └── pt-BR/ola.mdx
├── .hyper-down/                gerado — NÃO EDITAR
│   ├── content/article/
│   │   ├── types.ts            interface ambiente ArticleMeta
│   │   ├── builder.ts          articleRepository preguiçoso (só servidor)
│   │   └── modules.ts          import.meta.glob estático dos corpos MDX
│   └── default.ts              barrel contentModules
├── frontmatter.json            definições de tipos (formato Front Matter CMS)
├── hyperdown.config.json       contentDir, sitemap, i18n
└── vite.config.ts              hyperdownMdxPlugin → framework → hyperdownPlugin
```

Duas regras de fiação importam: o `hyperdownMdxPlugin()` **deve ser registrado antes
dos plugins do framework** (ele embrulha o `@mdx-js/rollup` e intercepta imports
`*.mdx?raw`, que o MDX compilaria mesmo com a query string), e a config de SSR mantém
`bun:sqlite`/`node:sqlite` externos enquanto bundleia o próprio pacote
(`noExternal: ["@indago/hyper-down"]`). Depois de um build, `dist/metadata/` contém os
`.db` finais — o "resultado" deployável do pipeline de conteúdo.

## A CLI

O binário `hyperdown` é instalado com o pacote. Os comandos perguntam interativamente
quando as flags são omitidas e são totalmente scriptáveis quando não:

| Comando              | Propósito                                                       |
| -------------------- | --------------------------------------------------------------- |
| `init [target]`      | Cria `hyperdown.config.json` / `frontmatter.json` / ambos.      |
| `validate [target]`  | Valida config e/ou frontmatter contra os schemas embutidos.     |
| `update schemas`     | Rebaixa os schemas do Front Matter CMS e regenera os tipos.     |
| `gen:db`             | Roda o codegen e escreve os bancos SQLite por tipo.             |
| `create-frontmatter` | Cria um `frontmatter.json` novo com tipos de conteúdo + i18n.   |
| `create-content`     | Adiciona um tipo de conteúdo a um `frontmatter.json` existente. |
| `create-item`        | Cria um novo item `.mdx` na pasta/idioma certos.                |

```bash
# Bootstrap
bunx @indago/hyper-down init both

# Novo tipo de conteúdo — o sufixo `req|opt` é obrigatório em cada campo
bunx @indago/hyper-down create-content --name product --folder Products \
  --fields "title:string:req,price:number:opt,status:choice[draft|live]:req"

# Novo item, depois regenere os bancos
bunx @indago/hyper-down create-item --type article --slug ola-mundo --lang pt-BR
bunx @indago/hyper-down gen:db

# Gate de CI — sai com código diferente de zero no primeiro arquivo inválido
bunx @indago/hyper-down validate both
```

O `gen:db` roda o codegen de `.hyper-down/**` antes do writer, então funciona num
checkout limpo, sem build prévio.

## O servidor MCP: agentes como autores de primeira classe

O pacote também traz o **`hyperdown-mcp`**, um servidor MCP stdio que expõe a CLI como
ferramentas: `hyperdown_init`, `hyperdown_validate`, `hyperdown_update`,
`hyperdown_gen_db`, `hyperdown_create_content`, `hyperdown_create_frontmatter` e
`hyperdown_create_item`. As ferramentas de criação exigem o conjunto completo de flags —
prompts interativos ficam desabilitados sob MCP.

**Claude Code** — adicione ao `.mcp.json` do projeto (ou rode
`claude mcp add hyperdown -- bunx --package @indago/hyper-down hyperdown-mcp`):

```json
{
  "mcpServers": {
    "hyperdown": { "command": "bunx", "args": ["hyperdown-mcp"] }
  }
}
```

**opencode** — declare como servidor local no `opencode.json`:

```json
{
  "$schema": "https://opencode.ai/config.json",
  "mcp": {
    "hyperdown": { "type": "local", "command": ["bunx", "hyperdown-mcp"], "enabled": true }
  }
}
```

Com qualquer um dos clientes, um agente consegue criar um tipo de conteúdo, criar itens
e regenerar os bancos de ponta a ponta. (Aliás, este próprio artigo foi criado com
`hyperdown_create_item`.)

## O pacote `.agents/`

O pacote npm embute uma árvore `.agents/` — material de referência para agentes de IA
trabalhando num repo que instala o HyperDown, dividido em **rules** (restrições) e
**skills** (receitas de tarefa):

- `rules/architecture.md` — os invariantes do pipeline: SSR-only, FTS5 contentless,
  repositório preguiçoso, ordem dos plugins, mapa de exports.
- `rules/configuration.md` — os formatos de `hyperdown.config.json` + `frontmatter.json`.
- `rules/checks.md` — checagens obrigatórias e arquivos gerados que não devem ser editados.
- `skills/cli.md`, `skills/add-content-item.md`, `skills/manage-content-types.md` —
  receitas passo a passo para os fluxos comuns de autoria.

Aponte seu `AGENTS.md`/`CLAUDE.md` para
`node_modules/@indago/hyper-down/.agents/README.md` e o agente herda o manual de
operação inteiro.

## Como ele é testado

- **O próprio motor** roda `bun test` — os testes de componente usam happy-dom +
  Testing Library; writer, repositório e codegen têm suítes unitárias.
- **Todo app gerado** traz as mesmas duas camadas: um teste Vitest de **integridade de
  conteúdo** (varre `content/`, garante que todo `.mdx` tem título + data válidos no
  frontmatter, nos dois idiomas) e uma suíte **e2e em Playwright** cobrindo artigos,
  receitas, projetos, busca full-text (uma query que casa mantém resultados; uma query
  sem sentido mostra o estado vazio), i18n e navegação.
- **O harness do scaffold** (`bun run test:templates`) empacota os motores como
  tarballs, gera cada template num diretório temporário, instala, builda, typecheca e
  roda unit + e2e — quatro frameworks contra um único conjunto de specs.

## Os scaffolds: quatro frameworks, um contrato

`bun create @indago/app` gera um app funcional já fiado ao HyperDown + HyperJson, na sua
escolha de **Vike**, **React Router v7**, **TanStack Start** ou **Next.js**. Todo
template traz as **mesmas rotas** (`/articles` com busca ao vivo, `/articles/:slug`,
`/cooking[/:slug]`, `/projects`, `/pt/*`) e as **mesmas suítes de teste** — o que muda
é como cada framework alimenta o HyperDown:

| Capacidade  | Vike            | React Router v7    | TanStack Start   | Next.js                        |
| ----------- | --------------- | ------------------ | ---------------- | ------------------------------ |
| Bundler     | Vite            | Vite               | Vite             | Webpack                        |
| Dados (SSR) | loaders `+data` | `loader` de rota   | `createServerFn` | Server Components              |
| Corpos MDX  | glob do Vite    | glob do Vite       | glob do Vite     | imports explícitos `@next/mdx` |
| Servidor    | Hono            | react-router-serve | srvx             | `next start`                   |

O caso fora da curva é o **Next.js**: sem Vite, o adapter `@indago/hyper-down/next`
(`withHyperDown` + `runHyperDownNextCodegen`) reescreve o `modules.ts` gerado em imports
explícitos do `@next/mdx` e roda o codegen no predev/prebuild — mesma arquitetura,
outra estratégia de resolução de módulos. Os três templates baseados em Vite diferem
principalmente no idioma do loader e no servidor de produção; a camada de dados, a
árvore de conteúdo e o pipeline de renderização são idênticos.

## Comece a construir em cinco minutos

O caminho mais rápido é o scaffold — um app completo e testado, já ligado ao HyperDown, em um
comando:

```bash
bun create @indago/app
```

Já tem um app? Adicione o HyperDown direto nele:

```bash
bunx @indago/hyper-down init both     # config + frontmatter
bunx @indago/hyper-down create-item --type article --slug hello-world --lang en
bunx @indago/hyper-down gen:db        # constrói o banco pesquisável
```

Esse é o loop inteiro: escreva em Markdown, indexe frontmatter **e** corpo num SQLite FTS5
contentless, consulte apenas no servidor e resolva o corpo MDX por um glob seguro para o
navegador — com CLI, servidor MCP, pacote `.agents/` e quatro scaffolds que honram o mesmo
contrato.

O código está no [GitHub](https://github.com/ZauJulio/indago), e se o seu problema é JSON
estruturado em vez de prosa, o irmão [HyperJson](/pt/articles/what-is-hyperjson) faz pelo
conteúdo tipado o que o HyperDown faz pela busca. Construa algo com ele — eu vou adorar ver o
que você criar.
