# O que é o HyperDown?

O **HyperDown** (`@muttum/hyper-down`) é um motor de conteúdo Markdown/MDX → SQLite para projetos Vite + React. Ele compila o **frontmatter** do seu conteúdo num índice **FTS5** SQLite compacto e contentless e o consulta **exclusivamente no servidor**, dentro dos loaders de rota. O resultado é busca full-text rápida e tipada sobre seus artigos e receitas — **sem banco de dados no cliente**, sem worker OPFS e sem `.db` enviado ao navegador.

É um dos dois motores independentes deste portfólio. O HyperDown cuida de **Markdown/MDX**; seu irmão [HyperJson](/pt/articles/what-is-hyperjson) cuida do **JSON estruturado**. Eles não compartilham código.

## A ideia central: metadados no SQLite, corpo nos globs

O HyperDown divide cada conteúdo em duas partes:

- **Os metadados do frontmatter** (título, tags, data, cozinha…) são parseados e guardados no SQLite. É contra isso que a busca e os filtros rodam.
- **O corpo MD/MDX nunca é guardado no SQLite.** Ele carrega em tempo de render via módulos estáticos `import.meta.glob`, gerados por codegen na árvore `.hyper-down/` do seu app.

As tabelas FTS5 usam `content=""` (contentless), então guardam apenas o índice invertido — não os valores originais dos campos. A coluna do corpo é **tokenizada no índice mas nunca persistida**, então a busca full-text alcança o texto do artigo enquanto o arquivo `.db` permanece minúsculo e o invariante "corpo nunca guardado" se mantém.

## SSR-only por design

O SQLite é lido somente no servidor, nos loaders `+data` do Vike, usando `bun:sqlite` (ou `node:sqlite` no Node ≥ 22 para plataformas como a Vercel). O navegador nunca vê um banco. A camada de dados é um `ContentRepository<T>` orientado a objetos exportado de `@muttum/hyper-down/server`, oferecendo:

- `search()` — FTS + filtros + ordenação + paginação
- `distinctValues()` — facetas para UIs de filtro
- `getMetaBySlug()` — metadados serializáveis com fallback de idioma

As páginas de listagem são **dirigidas pela URL**: o loader lê `q`/`tag`/`page`/`sort` da query string, roda o repositório e retorna resultados serializáveis. As páginas de detalhe carregam metadados via `getMetaBySlug` e resolvem o corpo MDX por um resolver seguro para o navegador — então o cliente SQLite é totalmente removido do bundle do cliente.

## Experimente pela CLI

```bash
# Cria config + frontmatter
hyperdown init both

# Cria um novo artigo (.mdx) num idioma
hyperdown create-item --type article --slug o-que-e-hyperdown --lang pt-BR

# Valida config + frontmatter
hyperdown validate both

# Gera os bancos SQLite a partir do seu conteúdo
hyperdown gen:db
```

Um **servidor MCP** (`hyperdown-mcp`) expõe `hyperdown_init`, `hyperdown_validate`, `hyperdown_update`, `hyperdown_create_frontmatter` e `hyperdown_create_item` — para que um agente de IA crie e escreva conteúdo de ponta a ponta. (Aliás, este próprio artigo foi criado com `hyperdown_create_item`.)

## O pipeline de build

O plugin do Vite (`hyperdownPlugin`) roda o codegen no `buildStart`: escreve os arquivos `{types,builder,modules}.ts` por tipo e o `.hyper-down/default.ts`, depois dispara o writer para emitir cada `.db`. No `closeBundle` ele copia cada `.db` para `dist/metadata/`, deixando o `dist` autocontido — um deploy SSR lê os bancos de lá sem precisar da árvore `content/` de origem.

Esse é o HyperDown: escreva em Markdown, busque no SQLite, e não envie nada extra ao cliente.
