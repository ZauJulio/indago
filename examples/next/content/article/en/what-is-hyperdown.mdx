# What is HyperDown?

**HyperDown** (`@virtus/hyper-down`) is a Markdown/MDX → SQLite content engine for Vite + React projects. It compiles your content's **frontmatter** into a compact, contentless **FTS5** SQLite index, then queries it **exclusively on the server** inside route loaders. The result is fast, typed, full-text search over your articles and recipes — with **no client-side database**, no OPFS worker, and no `.db` shipped to the browser.

It is one of two independent engines in this portfolio. HyperDown owns **Markdown/MDX**; its sibling [HyperJson](/articles/what-is-hyperjson) owns **structured JSON**. They share no code.

## The core idea: metadata in SQLite, body in globs

HyperDown splits each piece of content in two:

- **Frontmatter metadata** (title, tags, date, cuisine…) is parsed and stored in SQLite. This is what search and filters run against.
- **The MD/MDX body is never stored in SQLite.** It loads at render time through static `import.meta.glob` modules, codegen-generated into your app's `.hyper-down/` tree.

The FTS5 tables use `content=""` (contentless), so they store only the inverted index — not the original field values. The body column is **tokenized into the index but never persisted**, so full-text search reaches article prose while the `.db` file stays tiny and the "body never stored" invariant holds.

## SSR-only by design

SQLite is read only on the server, in Vike `+data` loaders, using `bun:sqlite` (or `node:sqlite` on Node ≥ 22 for platforms like Vercel). The browser never sees a database. The data layer is an OOP `ContentRepository<T>` exported from `@virtus/hyper-down/server`, offering:

- `search()` — FTS + filters + sort + pagination
- `distinctValues()` — facets for filter UIs
- `getMetaBySlug()` — serializable metadata with locale fallback

Listing pages are **URL-driven**: the loader reads `q`/`tag`/`page`/`sort` from the query string, runs the repository, and returns serializable results. Detail pages load metadata via `getMetaBySlug` and resolve the MDX body through a browser-safe content resolver — so the database client is fully stripped from the client bundle.

## Try it from the CLI

```bash
# Scaffold config + frontmatter
hyperdown init both

# Create a new article (.mdx) in a given locale
hyperdown create-item --type article --slug what-is-hyperdown --lang en

# Validate config + frontmatter
hyperdown validate both

# Generate the SQLite databases from your content
hyperdown gen:db
```

An **MCP server** (`hyperdown-mcp`) exposes `hyperdown_init`, `hyperdown_validate`, `hyperdown_update`, `hyperdown_create_frontmatter`, and `hyperdown_create_item` — so an AI agent can scaffold and author content end-to-end. (In fact, this very article was scaffolded with `hyperdown_create_item`.)

## The build pipeline

The Vite plugin (`hyperdownPlugin`) runs codegen on `buildStart`: it writes the per-type `{types,builder,modules}.ts` files and `.hyper-down/default.ts`, then spawns the writer to emit each `.db`. On `closeBundle` it copies every `.db` into `dist/metadata/` so the built `dist` is self-contained — an SSR deploy reads the databases there without needing the source `content/` tree.

That's HyperDown: author in Markdown, search in SQLite, ship nothing extra to the client.
